DNS — Domain Name System

What is DNS?

The Domain Name System is a hierarchical, distributed naming system that translates human-readable domain names (like YOUR_USERNAME.github.io) into machine-readable IP addresses. It functions as the internet's phonebook — without it, users would need to memorize numerical IP addresses for every website they visit.

How DNS Resolution Works for This Site

When a user visits this website, the following DNS resolution process occurs:

1
Browser Cache Check Browser checks its local DNS cache for a cached record
↓
2
Recursive Resolver Query sent to ISP's recursive DNS resolver (or 8.8.8.8 / 1.1.1.1)
↓
3
Root Nameserver Resolver queries root server, which directs to .io TLD nameserver
↓
4
TLD Nameserver .io TLD server directs to GitHub's authoritative nameservers
↓
5
Authoritative Response GitHub's DNS returns the A record (IPv4) and AAAA record (IPv6)

02

IP Addressing

IPv4 vs IPv6

IP addresses serve as unique identifiers for devices on a network. This website resolves to both IPv4 and IPv6 addresses, demonstrating dual-stack compatibility on GitHub's infrastructure.

IPv4
185.199.108.153

32-bit address — ~4.3 billion possible addresses. Written in dotted-decimal notation (four octets). GitHub Pages uses four IPv4 addresses for load balancing and redundancy.

IPv6
2606:50c0:8000::153

128-bit address — ~340 undecillion possible addresses. Written in hexadecimal colon notation. Solves IPv4 exhaustion and includes built-in security features (IPsec).

Note: The A record returns IPv4 addresses and the AAAA record returns IPv6 addresses. Multiple addresses indicate load balancing across GitHub's CDN (Fastly).
03

HTTP/HTTPS Protocols

HTTP vs HTTPS

HTTP (Hypertext Transfer Protocol) operates on port 80 and transmits data in plaintext. HTTPS (HTTP Secure) operates on port 443 and encrypts all communication using TLS (Transport Layer Security). This site enforces HTTPS — any HTTP request is automatically redirected with a 301 status code.

Key HTTP Headers

Header Value Purpose
content-type text/html; charset=utf-8 Specifies the media type of the response
server GitHub.com Identifies the web server software
x-github-request-id [unique ID] GitHub's internal request tracking
strict-transport-security max-age=31536000 HSTS — forces HTTPS for 1 year
x-content-type-options nosniff Prevents MIME-type sniffing attacks

Common Status Codes

200 OK Request succeeded, resource returned
301 Moved Permanent redirect (HTTP → HTTPS)
304 Not Modified Cached version is still valid
404 Not Found Requested resource doesn't exist
500 Server Error Internal server error

Note: The HTTP/2 200 confirms a successful HTTPS connection. The strict-transport-security header enforces HSTS, preventing downgrade attacks.
04

Security Implementation

HTTPS Enforcement

GitHub Pages provides automatic HTTPS via Let's Encrypt TLS certificates. This ensures all communication between the client and server is encrypted, protecting against man-in-the-middle (MITM) attacks, eavesdropping, and data tampering.

TLS Handshake Process

1
Client Hello Browser sends supported TLS versions and cipher suites
↓
2
Server Hello Server selects TLS 1.3 and cipher suite, sends certificate
↓
3
Certificate Verification Browser verifies certificate chain against trusted CAs
↓
4
Key Exchange Diffie-Hellman key exchange establishes shared session key
↓
5
Encrypted Session All subsequent data encrypted with AES-256-GCM symmetric encryption

Security Headers Present

✓
Strict-Transport-Security (HSTS)

Forces browsers to use HTTPS for all future requests for 1 year (31536000 seconds). Prevents protocol downgrade attacks.

✓
X-Content-Type-Options: nosniff

Prevents browsers from MIME-type sniffing, reducing risk of XSS attacks via file type confusion.

✓
X-Frame-Options: DENY

Prevents the site from being embedded in iframes, protecting against clickjacking attacks.
