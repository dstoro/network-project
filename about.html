Setup Steps

01
Repository Creation

Created a GitHub repository (dstoro/network-project) for version control and continuous deployment integration with Vercel.

02
Website Development

Built a three-page site using HTML5, CSS3, and vanilla JavaScript in VS Code. Dark theme with terminal-inspired UI and particle animations reflecting the networking/cybersecurity subject matter.

03
Vercel Deployment

Connected the GitHub repo to Vercel via GitHub integration. Every push to main triggers an automatic rebuild and deployment to Vercel's edge network. Site deployed at network-project-self.vercel.app.

04
HTTPS Enforcement

Vercel automatically provisions a Let's Encrypt TLS certificate and enforces HTTPS via 308 Permanent Redirect on all HTTP requests. HSTS header set with max-age=63072000 (~2 years).

05
Networking Verification

Ran nslookup, dig (A and AAAA queries), and curl -I (both HTTP and HTTPS) to gather real evidence of DNS resolution, IP addressing, HTTPS headers, and redirect enforcement.

Challenges & Solutions

Challenge DNS propagation delay — site not resolving immediately after initial deployment
Solution Waited for Vercel's DNS to propagate (~5 min). Used nslookup to verify resolution and confirmed the domain resolved to 216.198.79.67 and 64.29.17.67.
Challenge No IPv6 (AAAA) record returned for the Vercel subdomain
Solution Documented this honestly as a finding. Ran dig AAAA +short which returned no output, confirming IPv4-only. Explained that Vercel's default subdomains may not include IPv6 records, unlike custom domains.
Challenge Verifying that HTTPS is enforced, not just available
Solution Ran curl -I http://network-project-self.vercel.app (plain HTTP) and confirmed a 308 Permanent Redirect with Location: https://..., proving enforcement rather than just availability.
